基于您提供的详细文档（特别是《03_竞品分析与市场定位》和架构规划），以及当前 AI 学术工具的最新动态，我为您梳理了更全面的**竞品版图**和更具针对性的**评测体系**。

您的项目 ManuScript 核心定位是：**封闭域（Local/Private）** + **流水线式写作（Pipeline）** + **强引用溯源（Traceability）**。

### 第一部分：您的“假想敌”与真实竞品

虽然文档中已经列举了 Jenni AI 和 SciSpace，但从技术架构（RAG + Agent）和产品形态上看，您面临的竞争格局比想象中更复杂。我将其分为三类：

#### 1. 核心竞品（直接抢占用户场景）

这些产品直接解决“基于文档写论文”的问题，是您最大的威胁。

* **Google NotebookLM**（最大威胁）：
* **为何是竞品**：它完全是“封闭域 RAG”的标杆。用户上传 PDF，它基于这些 PDF 回答问题、生成摘要甚至播客。它的“Source Guide”功能与您的“引用溯源”非常像。
* **您的优势**：NotebookLM 更多是“对话/笔记”形态，缺乏**“大纲驱动的分节写作”**（它不能按论文结构一节节写）。您的“流水线（Assembly Line）”是它没有的。


* **Jenni AI / Paperpal**：
* **为何是竞品**：它们占据了“写作助手”的心智。Jenni 的自动续写非常流畅。
* **您的优势**：Jenni 的引用主要来自网络或简单的匹配，容易出现“引用不对文”的情况。您的 v2.0 架构强制每句话锚定 `[Doc_ID:Page]`，在**精准度**上可以降维打击。



#### 2. 技术竞品（架构相似，形态不同）

这些产品在技术路线上与您高度重合（Deep Research / Agentic RAG）。

* **OpenAI Deep Research / GPT-Researcher**：
* **区别**：它们是**开放域（Open Domain）**，在公网狂搜。您是**封闭域（Closed Domain）**，只在用户给定的 50 篇 PDF 里搜。
* **竞合关系**：用户通常需要先用 Elicit/Deep Research 找论文，下载下来，再扔给 ManuScript 写。您是它们的**下游**。


* **Dify / FastGPT (作为产品)**：
* 用户可以直接用 Dify 搭建一个“知识库问答助手”。
* **您的优势**：Dify 是通用的，它不懂“论文结构”，不懂“先写大纲再写正文”。您做的是**垂直场景的业务逻辑封装**。



#### 3. 潜在降维打击（大模型厂商的原生能力）

* **Claude Projects / GPTs**：
* 用户现在的确可以把 20 篇论文传给 Claude Project，然后说“帮我写”。
* **您的护城河**：Context Window 再大，一次性生成 5000 字长文依然会逻辑崩塌。您的**“Orchestrator-Worker”分节生成架构**，通过把大任务拆解为小任务，能突破 LLM 的注意力瓶颈。

### 第二部分：您的评测方向（Evaluation Strategy）

在架构迭代中，单纯凭“感觉”是不够的。对于 ManuScript v2.0/v3.0，建议建立一套**“LLM-as-a-Judge”**的自动化评测体系。

#### 1. 核心评测指标（Metrics）

您不需要测“文笔好不好”（这是主观的），您需要测“**逻辑准不准**”和“**引用对不对**”。

| 维度 | 指标名称 | 定义 | 评测方法 |
| --- | --- | --- | --- |
| **准确性 (Faithfulness)** | **引用幻觉率 (Citation Hallucination Rate)** | 引用的 `[Doc_1:12]` 里，原文真的支持生成的这句话吗？ | **最关键指标**。用 GPT-4o 当裁判，输入(原文片段, 生成句)，判断“Support/Contradict/Neutral”。 |
| **检索质量 (RAG Quality)** | **分节召回率 (Section Recall)** | 写“方法论”这一节时，RAGFlow 找回来的 Chunks 真的包含方法论描述吗？ | 人工标注 10 个典型章节的“金标准素材”，看 RAG 能召回多少。 |
| **Agent 智能度** | **规划合理性 (Planning Score)** | Planner Agent 生成的大纲和检索词，是否符合人类逻辑？ | 既然您有 v1.0 和 v2.0，对比两者生成的 Query 列表，看谁更精准覆盖主题。 |
| **用户体验** | **修正率 (Correction Rate)** | 用户拿到草稿后，修改了多少字？ | (原始字数 - 修改距离) / 原始字数。越低说明 AI 写得越不可用。 |

#### 2. 架构对比评测（A/B Testing）

既然您想“比较各个架构之间的差别”，您可以设计以下三组实验：

* **实验 A：RAG 效果对比**
* **对照组**：简单的向量检索（Baseline）。
* **实验组**：v2.0 中的 `Hybrid Search` (关键词+向量) + `Re-ranking`。
* **预期结论**：证明 RAGFlow 的混合检索在学术术语（专有名词）上表现更好。


* **实验 B：Agent 编排对比 (Single vs Multi)**
* **对照组 (v0.1)**：直接把素材丢给 LLM：“写这一节”。
* **实验组 (v2.0)**：Planner 分解 -> Query Generator 扩充 -> Ranker 筛选 -> Writer 写作。
* **预期结论**：Anthropic 的文章提到，多智能体系统在复杂任务上能提升 90% 的表现。您需要验证在“长难章节”上，多智能体是否减少了遗漏。


* **实验 C：Deep Research 模式 (v3.0 预研)**
* **测试**：当本地素材不足（Gap Detection）时，引入 Web Search 补充素材，对最终草稿的**信息丰富度**有多大提升？



#### 3. 如何落地评测（Vibe Coding 方式）

不要手动去测。您可以让 Claude Code 帮您写一个评测脚本：

> **Prompt 示例：**
> “我需要为一个 RAG 写作系统写一个评测脚本。
> 输入：一个 JSON 文件，包含 50 个样本，每个样本有 `(source_chunks, generated_text)`。
> 逻辑：使用 GPT-4o 作为裁判。对于每一句生成的文本，检查它是否被 `source_chunks` 里的内容所支撑。
> 输出：计算‘幻觉率’（没有支撑的句子比例）。
> 请使用 `ragas` 或 `deepeval` 库来实现。”

### 总结

* **竞品上看**：您的护城河不在于“能写”，而在于**“结构化地写”**（通过大纲分节）和**“负责任地写”**（强制引用）。这是 NotebookLM 和 ChatGPT 目前做不到的精细度。
* **评测上看**：紧盯着 **“引用准确性”** 和 **“分节相关性”** 这两个指标。如果 v2.0 的 Agent 编排能显著降低引用错误率，您的架构迭代就是成功的。
