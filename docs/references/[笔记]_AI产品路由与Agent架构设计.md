# 工作流与Agent架构设计：意图分类/路由层的通用性
- 进入工业级落地，兼顾考虑成本、速度等因素后，有个值得关注的现象！
- 路由分类->分而治之的模式，常见于80%以上的工业级AI产品落地
- ChatGPT默认模式也有个router
- 智能客服/AI问诊/AI教育等各种垂直应用
- 本质原因:这是AI大模型工业落地时,智能、成本、速度三者不可兼得的必然做法

业务复杂度与业务聚合性来决定是否需要路由


# 路由层之后的分支:关键是既要考虑业务特点,也要考虑大模型技术特点
- 路由层之后，后面应该分为几个分支？常见的分析策略
- 1. 要考虑业务天然的分支情况
- 2. 要考虑不同业务分支,在使用AI解决问题的时候,涉及的不同的技术要求
- 比如有的业务分支，需要做rag，有的不需要
- 比如有的业务分支,流程固化，可以用workflow;有的业务分支,天然就难以提前规划
- 案例: Al产品经理助手,包含通用知识回答,自动PRD、画图、深度研究、Jira测试case分析
- 思考路径：
- 1.首先判断不同的业务路径,是否存在“难度” "流程" "不确定性"的差异,如果是,需要做routing
- 2. routing该做。仔细分析,不同业务分支的“流程不确定性”、"工具组合”
- 通用知识问答，啥也不需要
- 画图模块，需要的工具 (可视化渲染)-套用evaluator-optimizor模式（渲染之后，再走一个截图。让另一个agent验证效果好不好，不好的话。要求重做)
- PRD模块，可能需要RAG
- 深度研究,最大的特点是不确定性很大,该模块必须用Agent (比如把claude agent sdk, open deep research拿来用) -再去套用ochestrator-worker模式




# 路由层之后的分支：关键是既要考虑业务特点，也要考虑大模型技术特点
- 路由层和核心的分支策略，是最重要的思考！！
- 以AI产品经理助手为例，PRD、深度研究、通用问答，宏观上怎么区分、怎么编排
- 不要被局部的业务的工作流/Agent编排,混淆了全局。
- 不能只盯着深度研究或PRD的细节

- 备注：这个思路偏产品化后期、比较全面的考虑。
- 在早期，不用过早考虑。早期可以抓住几个重点，多做一些不同的场景，比如单一LLM调用/单一大一统的Agent


# 工作流与Agent架构设计:一般怎么做?
适合绝大部分AI落地的架构演进步骤，以AI客服为例
1.先尝试单 Prompt 原型（验证可行性）-单一的LLM调用
2.加入RAG/工具接入（让答案“有依据、能查证、能办事”）- 增强的LLM调用
3.单场景 SOP 编排(把一个高频场景做成稳定流程) ,搭一个简单的workflow
4.路由 +策略层（多场景、多品类、多风险等级的治理） -搭一个复杂的工作流
- 先做分类路由，后面接N个不同的分支workflow
5.Agent循环 + 人在回路(不确定性高、需要追问/多步执行/升级) -在工作流中,看哪个环节适合引入Agent
通过体感和实测,从简单到workflow到Jagent,在建立很好的认知的情况下,最终拿出一个可能包含router,包含部分workflow部分agent的综合解决方案。
- 团队需要去体会,LLM在本业务下的表现的特点、Agent实际调用工具的不确定性的表现、工作流的时延,这些东西都有判断了,
- 再做复杂的面向终局的架构

# 训练营如何学习和实践Agent架构
- 观察：纯代码模式，不利于学习。
- 控制流与编排模块，Dify比Langgraph、Claude Code纯代码，学习效果好！因为可视化！
- 学习建议:前期偏学习状态,优先用Dify,利用Dify比较强大的能力,其实也能做出95%以上的核心的工作流&Agent的编排模式
- Dify编排模式本质上决定了：偏重控制流和编排31轻视记忆。
- Dify的重大不足1:上下文工程和记忆，很难自定义。
- Dify的不足:缺少最新的Agent工程的Harness环境（文件系统、bash命令) ,难以实现Claude Code或ExcelMaster 细分Coding超高难度Agent
学习建议：
- 1.前期优先学控制流与编排
- 优先用可视化的Dify,利用Dify比较强大的能力,其实也能做出95%以上的核心的工作流&Agent的编排模式
- 2. 在记忆模块、复杂的工具调用、Hook (Agent与传统代码的协作)之类的设计，优先纯代码，学习需要靠Langgraph
- 3.终局还是纯代码
- langgraph, deep agents, Claude code帮忙写代码实现
