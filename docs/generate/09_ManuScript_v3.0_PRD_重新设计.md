## 1. 执行摘要

### 1.1 项目背景

学术论文写作过程中，研究者面临的核心痛点：
- **论文多且长**：50-200 篇文献，英文为主，难以消化
- **记不住**：读后遗忘，想不起哪篇文章哪个观点
- **引用焦虑**：担心 AI 生成内容有幻觉，不敢直接使用
- **写作阻塞**：面对空白文档，不知如何下笔

### 1.2 产品定义

**ManuScript v3.0** 是一个**对话式文献研究助手**，帮助硕士研究生通过自然对话探索文献库，实现可追溯的检索与笔记管理。

**核心价值主张**：
> "让每一句话都能追溯到原文段落"

### 1.3 v3.0 核心变更

| 变更项 | v1.0-v2.0 | v3.0 (重新设计) |
|--------|------------|------------------|
| **核心定位** | 论文生成器 | 文献研究助手 |
| **交互方式** | 大纲驱动 | 对话驱动 |
| **处理流程** | 固定 6 步链 | 动态路由 |
| **主要输出** | 完整论文草稿 | 可追溯的检索结果 + 笔记 |
| **用户控制** | 黑盒生成 | 透明可验证 |

### 2.1 产品愿景

> "让每一位研究者都能快速找到文献中的任何内容，并验证其真实性"

### 2.3 核心隐喻

ManuScript 不是"聊天机器人"，而是 **"文献图书馆 + 研究助理"**：

| 角色 | 职责 | 对应功能 |
|------|------|---------|
| **图书馆** | 存储所有论文，支持精准检索 | 全文检索 + 段落索引 |
| **研究助理** | 理解用户需求，帮助整理素材 | 对话式探索 + 笔记管理 |
| **事实核查员** | 验证每句话的来源 | 溯源展示 + 引用验证 |

### 2.4 我们是什么 vs 我们不是什么

| ✅ 我们是 | ❌ 我们不是 |
|----------|------------|
| 文献研究与检索工具 | 论文一键生成器 |
| 可追溯的信息来源系统 | 黑盒 AI 生成器 |
| 对话式探索助手 | 聊天机器人 |
| 笔记管理与知识库 | 完整的写作工具 |
| 本地优先的隐私方案 | 云端全托管服务 |

```
├─────────────────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                                     │
│  年龄：25-30 岁                                                                                     │
│  场景：撰写毕业论文/期刊论文                                                                          │
│  文献量：已有 50-200 篇相关文献                                                                       │
│                                                                                                     │
│  痛点：                                                                                             │
│  • 文献太多，读了后面忘前面                                                                           │
│  • 不知道如何组织成连贯的论述                                                                         │
│  • 担心引用出错被导师批评                                                                             │
│  • 英文文献阅读效率低                                                                                 │
│                                                                                                     │
│  期望：                                                                                              │
│  • 快速找到"哪篇文章哪个段落说过什么"                                                                  │
│  • 验证生成内容的真实性                                                                               │
│  • 通过对话渐进式理解文献                                                                             │
│  • 管理阅读笔记和灵感                                                                                 │
│                                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 用户旅程

基于多轮讨论确认的用户旅程是**可回退的状态机**，而非线性流程：

**核心思想**：用户不是按固定顺序走完就结束，而是可以在各个阶段之间自由切换。

```
用户旅程状态机

┌─────────────────────────────────────────────────────────────────────────────────────┐
│                                                                                     │
│    探索态 ───────► 结构化态 ───────► 写作态 ───────► 润色态                           │
│     ▲                ▲                  ▲                  ▲                        │
│     │                │                  │                  │                        │
│     └────────────────┴──────────────────┴──────────────────┴──────────────────┘     │
│                                                                                     │
│   任意时刻都可以：                                                                    │
│   • 向前推进（探索 → 结构化 → 写作 → 润色）                                            │
│   • 向后退回（润色 → 写作 → 结构化 → 探索）                                            │
│   • 跨阶段跳跃（如：写作态 → 探索态）                                                  │
│                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘

典型场景示例：

场景1：正常前进
  探索态："我有哪些关于 CNN 的论文？"
    → 结构化态："这些论文可以怎么组织？"
    → 写作态："帮我写引言部分"

场景2：回退探索
  写作态："引言还缺少一篇对比方法的论文"
    → 探索态："帮我找一下关于 Transformer 在医学影像的应用"

场景3：跨阶段跳转
  润色态："参考文献格式有问题"
    → 结构化态："重新整理引用格式"
```

### 3.3 核心痛点分析

| 痛点 | 用户影响 | ManuScript 解法 |
|------|----------|-----------------|
| **认知过载** | 50 篇论文无法全部记住 | Section Context Window：每次只展示相关片段 |
| **引用焦虑** | 担心 AI 胡说八道 | Evidence Traceability：每句话都可追溯到原文段落 |
| **结构性卡顿** | 不知从何开始 | Draft as Verification：先展示素材，后生成内容 |
| **阅读遗忘** | 读后忘前 | 笔记入检索：用户笔记和论文一起可搜索 |

## 5. 核心功能规格

### 5.1 F1: 论文导入与解析

#### 5.1.1 功能描述

用户上传本地 PDF 或提供 arXiv 链接，系统解析为结构化 Markdown。

#### 5.1.2 数据获取优先级

| 优先级 | 来源 | 方式 | 覆盖范围 |
|--------|------|------|----------|
| **P0** | 本地上传 PDF | 用户拖拽/选择文件 | 100% |
| **P0** | arXiv 链接下载 | 系统自动下载 | arXiv 论文 |
| **P1** | arXiv API 搜索 | 集成搜索接口 | arXiv 全部 |
| **P1** | OpenAlex API | 元数据获取 | 广泛覆盖 |

#### 5.1.3 解析方案

| 方案 | 选择 | 说明 |
|------|------|------|
| **MVP** | Marker + PyMuPDF | Marker处理学术PDF（表格/公式/多栏），PyMuPDF做fallback |
| **P1** | arXiv LaTeX 优先 | pylatexenc解析，保留公式结构 |
| **兜底** | 纯文本 + OCR | Tesseract处理扫描件 |

#### 5.1.4 解析标准（最低可用）

| 内容类型 | 最低要求 | 降级策略 |
|----------|----------|----------|
| **章节结构** | 正确识别 Abstract/Introduction/Methods/Results/Discussion | 降级为大段落 |
| **段落** | 正确分段 | 人工分段 |
| **引用** | 提取引用标记 | 保留原始文本 |
| **表格** | 转为 Markdown 或描述性文字 | 纯文本占位 |
| **公式** | LaTeX 保留或描述 | 纯文本占位 |

#### 5.1.5 解析失败兜底策略

```
┌─────────────────────────────────────────────────────────────────────────────────────────────────────┐
│                              解析失败兜底策略                                                        │
├─────────────────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                                     │
│   完美解析                           │   部分失败                          │   完全失败               │
│   ───────────────────────────────────────────────────────────────────────────────────────────       │
│   结构完整                           │   结构混乱                          │   无法提取               │
│   │                                 │   │                                 │   │                     │
│   │                                 │   ▼                                 │   ▼                     │
│   │                                 │   保留纯文本                         │   告知用户               │
│   │                                 │   人工分段                           │   让用户重试/手动        │
│   │                                 │   允许全文检索                       │                         │
│   │                                 │                                     │                         │
│   ▼                                 ▼                                     ▼                         │
│   正常使用                           │   降级使用                          │   排除/警告              │
│                                                                                                     │
│   原则：宁可降级也不要丢数据                                                                          │
│                                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

### 5.2 F2: 全文检索与溯源

#### 5.2.1 功能描述

用户通过自然语言或搜索词查询文献，系统返回匹配的段落，并可点击追溯到原文。

#### 5.2.2 MVP 检索方案

| 方案 | 技术 | 性能 | 复杂度 |
|------|------|------|--------|
| **P0** | ripgrep 全文检索 + 段落索引 | < 2 秒 | 低 |
| P1 | SQLite FTS5 倒排索引 | < 0.5 秒 | 中 |
| P2 | 向量索引 + 全文融合 | < 1 秒 | 高 |

#### 5.2.3 检索输入方式

支持两种方式，系统自动识别：
- **搜索词**：直接输入关键词（如 "CNN 医学影像"）
- **自然语言**：完整提问（如 "有哪些论文讲了 CNN 在医学影像的应用？"）

#### 5.2.4 检索结果展示

```
┌─────────────────────────────────────────────────────────────────────────────────────────────────────┐
│                              检索结果展示                                                            │
├─────────────────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                                     │
│   📄 [1] Deep Learning for Medical Image Diagnosis                                                  │
│      Smith et al., 2024 | arXiv:2401.12345 | 来源: 原文                                              │
│      ───────────────────────────────────────────────────────────────────────────────────────────    │
│      Convolutional Neural Networks (CNNs) have emerged as a powerful tool for medical image         │
│      analysis. In chest X-ray interpretation, CNN-based models achieved accuracy comparable         │
│      to human experts...                                                                            │
│      ───────────────────────────────────────────────────────────────────────────────────────────    │
│      [📍 查看原文] [📝 添加笔记] [📋 复制引用]                                                       │
│                                                                                                     │
│   📝 [2] 笔记: CNN 核心特点                                                                          │
│      我的笔记 | 2024-01-15 | 来源: 笔记                                                               │
```

#### 5.2.5 来源区分策略（三重区分）

| 区分方式 | 原文 | 笔记 |
|----------|------|------|
| **图标** | 📄 蓝色 | 📝 黄色 |
| **标签** | "来源: 原文" | "来源: 笔记" |
| **背景** | 灰色背景条 | 黄色背景条 |

#### 5.2.6 排序策略

| 因子 | MVP 处理 | 未来增强 |
|------|---------|----------|
| 关键词匹配度 | 匹配数量 | TF-IDF 加权 |
| 来源优先级 | 原文 > 笔记（可切换） | 用户可配置 |
| 时间新鲜度 | 不考虑 | 按年份排序 |

### 5.5 F5: 会话管理

#### 5.5.1 多会话策略（已确认）

| 策略 | 选择 | 理由 |
|------|------|------|
| **会话数量** | 软限制 3 个 | 超过后提示用户关闭旧会话 |
| **状态持久化** | 恢复活跃会话，其他需手动恢复 | 平衡复杂度 |
| **并行处理** | 非活跃会话暂停，不消耗 token | 节省成本 |

#### 5.5.2 会话共享记忆（三层策略）

| 层级 | 内容 | 示例 |
|------|------|------|
| **全局共享** | 论文库元数据、用户笔记、项目状态、用户偏好 | 所有会话可见 |
| **会话独立** | 对话历史、临时搜索结果、会话级上下文 | 不共享 |
| **可引用** | 会话摘要、关键事实 | "基于会话1的结论..." |

#### 5.5.3 对话压缩触发策略

| 触发条件 | 操作 | 压缩内容写入 |
|----------|------|-------------|
| Token 接近上限 (>80%) | 自动压缩 | 会话摘要 + 关键结论 → 长期记忆 |
| 用户明确请求 | 立即压缩 | 结构化总结 → 长期记忆 |
| 会话结束时 | 生成全局摘要 | 会话总结 → 可被其他会话引用 |

```
用户查询

        ┌───────────────┼───────────────┐
        ↓               ↓               ↓
┌─────────────┐  ┌─────────────┐  ┌─────────────┐

       └────────────────┼────────────────┘
```

## 9. 性能与成本

### 9.1 性能目标（已确认）

| 操作类型 | MVP 目标 | 用户感知 |
|----------|---------|----------|

| 点击溯源（原文展开） | < 0.5 秒 | 即时响应 |
| 笔记保存 | < 0.3 秒 | 无感知 |

### 9.2 数据量级

| 指标 | 数值 | 说明 |
|------|------|------|
| 论文数量 | 50-200 篇 | 目标用户典型量级 |
| 存储空间 | ~2GB | 平均每篇 5000 字 + 元数据 |
| 检索响应 | < 2 秒 | ripgrep 全文检索 |

### 9.3 Token 消耗估算

| 场景 | 平均 Token 数 | 成本估算 |
|------|--------------|----------|
| 单次检索 | 500-1000 | 低 |
| 自然语言理解 | 1000-2000 | 中 |
| 会话摘要生成 | 2000-3000 | 中 |
| 完整对话轮次（10轮） | 10000-20000 | 中高 |

### 9.4 Token 显示策略（已确认）

**选择 B**：显示当前会话消耗

- 在状态栏显示当前会话 Token 数
- 让用户感知成本
- MVP 不做严格配额限制

### 9.5 异步任务策略（已确认）

**选择 B**：解析异步，其它同步

| 操作类型 | MVP 处理方式 | 未来增强 |
|----------|--------------|----------|
| 论文上传 + 解析 | **异步** | 后台任务队列 |
| 全文检索 | 同步 | 异步 + 流式返回 |
| 索引更新 | 同步 | 后台异步 |
| 会话摘要生成 | 同步（显示加载） | 后台任务 + 通知 |

### 9.6 硬件要求

| 配置 | 要求 | 体验说明 |
|------|------|----------|
| **最低** | 8GB内存，无显卡 | 解析较慢，Embedding建议用API |
| **推荐** | 16GB内存，4GB显卡 | 全功能本地运行 |

## 10. 隐私与安全

### 10.1 隐私模式（已确认）

**选择 B**：本地模式 + 云端开关

| 数据类型 | 存储位置 | 云端处理 |
|----------|---------|----------|
| 原始 PDF | 本地 | 不上传 |
| 解析后的 Markdown | 本地 | 需用户同意 |
| 用户笔记 | 本地 | 不上传 |
| 检索索引 | 本地 | 不上传 |
| 对话历史 | 本地 | 需用户同意 |

### 10.2 用户控制

- ✅ 本地优先模式（所有数据不上传）
- ☐ 脱敏后云端处理（需用户选择）
- ☐ 显示当前数据处理位置（本地/云端）

### 10.3 降级策略

| 失败场景 | 降级方案 | 用户体验 |
|----------|---------|----------|
| PDF 解析失败 | 保留纯文本版本 | 警告但继续使用 |
| arXiv 下载失败 | 提示用户手动上传 | 友好错误提示 |
| LLM API 不可用 | 切换到本地检索 | 显示"离线模式" |
| 检索无结果 | 提示调整关键词 | 搜索建议 |
| 索引损坏 | 重建索引 | 后台重建，通知用户 |
| **GPU 不可用（本地模式）** | **切换到 API 模式或 CPU** | **提示模式已切换** |
| **Embedding 失败（本地）** | **自动降级到 API 模式** | **静默切换，日志记录** |
| **ChromaDB 损坏** | **降级到 ripgrep + FTS5** | **显示"语义检索暂时不可用"** |
| **向量维度不匹配** | **重建索引或提示用户** | **友好错误提示** |

### 10.4 数据备份（已确认）

- 支持导出所有论文和笔记

## 附录 B：产品设计原则

1. **决策流程可收束**: 每轮最多 3 个选项、只问核心问题，避免用户在复杂选择中崩溃
2. **可信度优先**: 检索命中率 > 溯源准确性 > 检索速度 > 使用频率
3. **本地优先**: 默认本地处理，云端需用户显式同意
4. **渐进增强**: MVP 先做核心功能，高级功能作为 P1/P2 展示规划
5. **永不丢数据**: 降级策略确保用户数据安全
