## 2. 核心设计原则

### 2.1 产品原则

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        ManuScript 产品原则                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 【聚焦原则】                                                       │
│      识别核心价值，外包非核心能力                                       │
│      → 我们的价值是"写作流程"，不是"PDF 解析"                          │
│                                                                         │
│   2. 【场景匹配原则】                                                   │
│      技术方案要匹配具体场景，而非追逐热点                               │
│      → Deep Research 适合探索，Material Assembly 适合学术写作           │
│                                                                         │
│   3. 【信任优先原则】                                                   │
│      高风险场景（学术、医疗、金融）中，信任度 > 效率                    │
│      → 宁愿多一步确认，也不要黑盒输出                                   │
│                                                                         │
│   4. 【可控性原则】                                                     │
│      AI 工具应增强用户能力，而非替代用户决策                            │
│      → 用户确认每一步，而非一键生成                                     │
│                                                                         │
│   5. 【简单性原则】                                                     │
│      能用简单方案解决的问题，不要用复杂方案                             │
│      → 如果 v0.2 够用，就不需要跳到 v2.0                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 技术原则

| 原则 | 说明 | 应用场景 |
|------|------|---------|
| **最小可验证** | 每个版本聚焦验证特定假设 | 版本规划 |
| **快速迭代** | 先跑通再优化，不追求完美 | 开发节奏 |
| **可观测** | 每个环节都要可调试可观测 | Agent 设计 |
| **容错性** | 单点故障不影响整体 | 错误处理 |

## 3. 关键技术决策

> 以下是项目开始前已明确的关键技术决策。

### 3.1 自建 RAG vs RAGFlow 托管

**决策**：使用 RAGFlow 托管

| 维度 | 自建 RAG | RAGFlow 托管 |
|------|---------|-------------|
| PDF 解析质量 | 60-70%（需优化） | 90%+ |
| 工程投入 | 高（3-6个月） | 低（API对接） |
| 维护成本 | 持续投入 | 几乎无 |
| 灵活性 | 完全可控 | 受限于 API |

**决策理由**：
1. PDF 解析是"已解决的问题"，不是我们的核心价值
2. RAGFlow 已投入大量工程优化，我们无法超越
3. 节省 60% 工程量，聚焦写作流程创新

### 3.2 Deep Research vs Material Assembly

**决策**：以 Material Assembly 为核心

| 维度 | Deep Research | Material Assembly | 学术需求 |
|------|--------------|-------------------|---------|
| 信息源 | 开放网络 | 用户文献库 | 需可控来源 ✅ |
| 幻觉率 | 高（30-70%） | 低（<5%） | 零容忍 ✅ |
| 引用可靠性 | 不确定 | 100% 可溯源 | 必须准确 ✅ |
| 用户控制 | 低（黑盒） | 高（全程可见） | 人在回路 ✅ |

**决策理由**：学术写作者通常已读了大量论文，问题不是"信息不够"，而是"信息太多，组织不起来"

### 3.3 一键生成 vs 分节渐进

**决策**：分节渐进生成

| 维度 | 一键生成全文 | 分节渐进生成 |
|------|-------------|-------------|
| 用户体验 | 看起来很酷 | 需要多次操作 |
| 可控性 | 极低 | 高 |
| 幻觉风险 | 极高 | 低 |
| 用户心理负担 | 高（要核对5000字） | 低（逐段确认） |

**决策理由**：学术写作中，"控制感"比"便捷性"更重要

### 3.4 Agent 架构选型

**决策**：从简单到复杂，渐进式选型

| 版本 | 架构 | 选型理由 |
|------|------|---------|
| v0.1 | 单一 Prompt | 最快验证 RAGFlow 连通性 |
| v0.2 | Prompt Chain | 验证分节生成可行性 |
| v1.0 | Chain-of-Agents | 验证专业化 Agent 效果 |
| v2.0 | Orchestrator-Worker | 如需并行和灵活调度 |

**原则**：不预设最终架构，根据实际问题决定是否升级

### 4.2 实现目标

- [x] 能够生成带引用的文本

- [x] 中文输出 + 专业术语英文
- [x] References 显示被引用的 chunks 及内容摘要

### 4.3 遇到的问题

#### 问题 1：RAGFlow 返回字段名不匹配

```
【现象描述】
检索结果中 document_name 显示为 "Unknown"，无法正确显示文档名称。

【原因分析】
RAGFlow API 返回的字段名与代码中预期的不同：
- 预期：document_name, score
- 实际：document_keyword, similarity

【解决方案】
修改 prototype.py 中的字段映射：
doc_name = chunk.get("document_keyword") or chunk.get("document_name", "Unknown source")
score = chunk.get("similarity") or chunk.get("score", 0)

【经验教训】
调用第三方 API 时，应先调试确认实际返回的数据结构，而非依赖文档假设。
```

#### 问题 2：引用编号超出范围（引用幻觉）

```
【现象描述】
生成的文本中出现 [28] 等引用编号，但 References 只显示 5 个文档，
或者出现超出检索 chunks 数量的编号。

【原因分析】
1. LLM 在生成时可能"编造"不存在的引用编号
2. Prompt 中未明确限制引用编号范围

【解决方案】
1. 修改 Prompt，明确限制：**只能使用 [1] 到 [{num_chunks}] 的编号**
2. 将实际检索到的 chunks 数量传递给生成函数
3. 在输出时只显示被引用的 chunks（解析生成文本中的 [数字] 模式）

【经验教训】
LLM 生成引用时容易产生幻觉，必须在 Prompt 中明确约束，并在后处理中验证。
```

#### 问题 3：输出语言不匹配

```
【现象描述】
用户输入中文主题，但输出为英文段落。

【原因分析】
原始 Prompt 使用英文编写，未指定输出语言。

【解决方案】
修改 Prompt 为中文，并明确要求：
- 输出语言为中文
- 专业术语保留英文（如 Deep Research, RAG, LLM 等）

【经验教训】
Prompt 的语言和输出要求需要明确指定，不能依赖 LLM 自动推断。
```

#### 问题 4：References 格式问题

```
【现象描述】
1. References 列表不换行，显示混乱
2. 显示全部检索到的 chunks，而非只显示被引用的
3. 来源文档列表格式不清晰

【原因分析】
输出格式设计不够细致，未考虑 Markdown 渲染效果。

【解决方案】
1. 每个引用后加 \n\n 确保换行
2. 解析生成文本中的引用编号，只显示被引用的 chunks
3. 为每个引用添加内容摘要（前 100 字符）
4. 来源文档使用数字列表格式

【经验教训】
输出格式需要考虑实际渲染效果，信息展示要精简且有用。
```

#### 问题 5：RAGFlow API 连接超时

```
【现象描述】
Gradio UI 中点击生成后报错：httpx.ConnectTimeout

【原因分析】
1. 网络波动导致连接超时
2. 原超时设置为 30 秒，可能不够

【解决方案】
1. 将超时时间从 30s 增加到 60s
2. 添加更友好的错误提示：raise Exception("RAGFlow API 连接超时，请检查网络或稍后重试")

【经验教训】
网络请求需要合理的超时设置和友好的错误处理。
```

### 4.7 升级决策：v0.1 → v0.2

```
【是否升级】
- [ ] 继续升级到 v0.2
- [x] 跳过 v0.2，直接进入 v1.0
- [ ] 需要先解决某些问题

【v0.1 暴露的问题】
1. 单一 Prompt 难以处理复杂的多步骤任务 → v1.0 使用专业化 Agent 解决
2. 检索质量依赖单一查询 → v1.0 使用 Query Generator Agent 生成多个查询
3. 无法对检索结果进行排序筛选 → v1.0 使用 Chunk Ranker Agent 解决
4. 生成内容无法验证 → v1.0 使用 Verifier Agent 解决

【v0.1 vs v0.2 的区别】
┌──────────────┬─────────────────────┬─────────────────────┐
│ 维度         │ v0.1 最小原型        │ v0.2 基础流程        │
├──────────────┼─────────────────────┼─────────────────────┤
│ 核心目标     │ 验证 RAGFlow 连通性  │ 验证分节生成可行性   │
│ Query 策略   │ 单一查询（直接输入）  │ Query 生成优化       │
│ 引用处理     │ 基础引用标记         │ 引用格式化逻辑       │
│ 大纲管理     │ 无                   │ JSON 结构定义        │
│ 批量处理     │ 单章节               │ 多章节批量           │
└──────────────┴─────────────────────┴─────────────────────┘

【为什么跳过 v0.2】

1. 功能重叠
   v0.2 计划实现的核心功能在 v1.0 中已以 Agent 形式实现：
   • Query 生成 → v1.0 的 Query Generator Agent
   • 引用格式化 → v1.0 的 Draft Writer Agent + Verifier Agent

2. 增量价值不足
   v0.2 相对于 v0.1 是"渐进式优化"而非"架构升级"：
   • 单独做 v0.2 会产生重复开发工作（代码在 v1.0 需重写）
   • 额外的测试成本
   • 延迟验证 Agent 架构的时间

3. v0.1 已验证核心假设
   • RAGFlow API 可用（检索到 19-28 个 chunks）
   • LLM 生成质量可接受
   • 引用可追溯
   → 可以直接跳到更复杂的架构验证

4. 问题驱动原则
   v0.1 暴露的问题需要 Agent 级别的解决方案，而非 v0.2 的 Prompt 优化：

   v0.1 问题                        解决方案
   ─────────────────────────────────────────────────
   单一 Prompt 难以处理复杂任务  →  v1.0 专业化 Agent
   检索质量依赖单一查询          →  Query Generator Agent
   无法对检索结果排序筛选        →  Chunk Ranker Agent
   生成内容无法验证              →  Verifier Agent

【决策结论】
跳过 v0.2，直接进入 v1.0 测试。

理由：
• v0.1 已验证核心技术可行性
• v0.2 的增量改进不足以单独成为一个版本
• v0.2 的功能已被 v1.0 覆盖
• 符合"简单性原则"：如果问题本身需要复杂方案，就不要在中间版本浪费时间
```

### 5.1 不采用原因

v0.2 原计划作为 v0.1 到 v1.0 的过渡版本，代码已完成但经过评估后决定不采用。

**已完成的代码（v0_2/ 目录）：**
- `pipeline.py` - 基础流程管道（~10KB）
- `query_generator.py` - Query 生成逻辑（~4KB）
- `citation_formatter.py` - 引用格式化（~5KB）
- `main.py` - Gradio UI（~8KB）

**原计划实现的功能：**
- 大纲 JSON 结构定义
- Query 生成 Prompt 调优
- 引用格式化逻辑
- 多章节批量处理

**不采用的核心原因：**

| 原因 | 说明 |
|------|------|
| **功能重叠** | v0.2 的 Query 生成、引用格式化已在 v1.0 以 Agent 形式实现 |
| **增量价值不足** | 渐进式优化不足以单独成为一个版本 |
| **问题需要架构升级** | v0.1 暴露的问题需要 Agent 级别的解决方案 |
| **敢于放弃沉没成本** | 代码已写完，但识别到继续使用会产生技术债 |

### 5.2 v0.2 与 v1.0 的功能对应

```
v0.2 计划功能              v1.0 实现方式
────────────────────────────────────────────────
Query 生成优化         →   Query Generator Agent
引用格式化逻辑         →   Draft Writer Agent
多章节批量处理         →   Section Planner Agent
大纲 JSON 结构         →   Workflow State 管理
```

### 5.3 决策反思

```
【这个决策的价值】
写完代码后主动放弃，比"跳过不写"更能体现决策能力：
• 证明了对问题的深入理解（写完才发现不够用）
• 展示了敢于放弃沉没成本的魄力
• 代码保留可作为对比参考

【评估标准】
• v1.0 开发是否顺利
• 是否因不采用 v0.2 而遗漏某些基础验证
• Agent 架构是否真的比 Prompt Chain 更好
```
