## 1. 项目介绍话术

### 1.1 30 秒版本（电梯演讲）

> ManuScript 是一个 AI 辅助的学术论文写作工具。它的核心价值是帮助研究者基于自己的文献库，生成带准确引用的论文段落。
>
> 与 Deep Research 类工具不同，ManuScript 采用 Material Assembly 模式——不是去网上搜索信息，而是组织用户已有的文献素材。这样做的好处是引用 100% 可追溯，避免了学术写作中最致命的"引用幻觉"问题。
>
> 技术上，我采用了渐进式架构演进，从最小原型验证到 Agent 链编排，每个版本都有明确的验证目标。

### 1.2 1 分钟版本

> ManuScript 是我独立开发的 AI 学术写作辅助工具。
>
> **解决的问题**：研究者在写论文时，往往已经读了大量文献，但组织这些素材写成连贯的段落很耗时。现有的 AI 写作工具要么容易产生幻觉，要么引用不可追溯。
>
> **我的方案**：采用 Material Assembly 模式，用户上传自己的文献库，系统通过 RAG 检索相关内容，然后生成带准确引用的段落。每个引用都可以追溯到原文，用户可以逐段确认。
>
> **技术实现**：
> - 使用 RAGFlow 作为 RAG 引擎，处理 PDF 解析和向量检索
> - 使用 OpenAI API 进行文本生成
> - 采用渐进式架构演进：v0.1 验证连通性 → v1.0 引入 Agent 链 → v2.0 动态调度
>
> **关键成果**：v0.1 原型已跑通，检索到 19-28 个相关 chunks，引用准确率良好。目前正在进行 v1.0 的 Agent 架构测试。

### 1.3 3 分钟版本（详细展开）

> **项目背景**
>
> 我在准备毕业论文时发现一个痛点：我已经读了几十篇论文，但把这些内容组织成连贯的文献综述非常耗时。市面上的 AI 写作工具有两个问题：一是容易产生幻觉，二是引用不可追溯。对于学术写作来说，这两点是致命的。
>
> **产品定位**
>
> ManuScript 的核心理念是 Material Assembly（素材组装），而非 Deep Research（深度搜索）。区别在于：
> - Deep Research 从开放网络搜索信息，幻觉率高达 30-70%
> - Material Assembly 只使用用户提供的文献，引用 100% 可追溯
>
> 这个定位来自一个洞察：学术写作者的问题不是"信息不够"，而是"信息太多，组织不起来"。
>
> **技术架构**
>
> 我采用了渐进式架构演进策略：
>
> 1. **v0.1 最小原型**：验证 RAGFlow API 连通性，单一 Prompt 生成
>    - 成果：成功检索 19-28 个 chunks，生成质量可接受
>    - 问题：单一 Prompt 难以处理复杂任务，检索质量依赖单一查询
>
> 2. **v0.2 基础流程** `[代码完成未采用]`：原计划做 Prompt Chain 优化
>    - 计划功能：Query 生成优化、引用格式化、多章节批量处理
>    - 代码已完成（v0_2/ 目录，约 600 行）
>    - 不采用原因：功能与 v1.0 重叠，写完后发现增量价值不足
>    - 这个决策体现了"敢于放弃沉没成本"的能力
>
> 3. **v1.0 Chain-of-Agents**：引入专业化 Agent
>    - Query Generator：生成多个检索查询
>    - Chunk Ranker：对检索结果排序筛选
>    - Draft Writer：生成带引用的段落
>    - Verifier：验证引用准确性
>
> 4. **v2.0 Orchestrator-Worker**：动态调度，支持并行处理
>
> 5. **v3.0 混合模式** `[待开发]`：本地文献 + 网络搜索补充
>
> **关键决策**
>
> 一个重要的架构决策是写完 v0.2 后决定不采用。我完成了 v0.2 的代码（约 600 行），但分析后发现：
> - v0.2 的功能（Query 生成、引用格式化）在 v1.0 已以 Agent 形式实现
> - v0.1 暴露的问题需要架构级别的解决方案，而非 Prompt 优化
> - 继续使用 v0.2 会产生技术债，代码在 v1.0 需要重写
>
> 这个决策体现了"敢于放弃沉没成本"：写完代码后主动放弃，比"跳过不写"更能证明对问题的深入理解。
>
> **当前进展**
>
> v0.1 已完成测试，v1.0 代码已完成待测试。整个项目展示了从 0 到 1 的架构演进能力，以及基于真实数据做技术决策的方法论。

## 2. 关键技术决策

### 2.1 决策一：RAGFlow 托管 vs 自建 RAG

**决策**：使用 RAGFlow 托管

**背景**：
项目需要处理 PDF 文献的解析和向量检索。有两个选择：
1. 自建 RAG 系统（LangChain + Chroma + PDF 解析库）
2. 使用 RAGFlow 托管服务

**分析过程**：

| 维度 | 自建 RAG | RAGFlow 托管 |
|------|---------|-------------|
| PDF 解析质量 | 60-70%（需大量优化） | 90%+（已优化） |
| 工程投入 | 高（3-6个月） | 低（API 对接） |
| 维护成本 | 持续投入 | 几乎无 |
| 灵活性 | 完全可控 | 受限于 API |

**决策理由**：
1. PDF 解析是"已解决的问题"，不是我们的核心价值
2. RAGFlow 已投入大量工程优化表格、公式、图片的解析
3. 节省 60% 工程量，聚焦写作流程创新

**面试话术**：
> "我选择 RAGFlow 而非自建 RAG，是因为识别到 PDF 解析不是我的核心价值。RAGFlow 在这个领域已经投入了大量工程优化，我无法超越也没必要重复。这让我能把精力放在真正的创新点——写作流程的 Agent 编排上。"

---

### 2.2 决策二：Deep Research vs Material Assembly

**决策**：以 Material Assembly 为核心

**背景**：
AI 辅助写作有两种模式：
1. Deep Research：从网络搜索信息，自动生成内容
2. Material Assembly：基于用户提供的素材，组织生成内容

**分析过程**：

| 维度 | Deep Research | Material Assembly | 学术需求 |
|------|--------------|-------------------|---------|
| 信息源 | 开放网络 | 用户文献库 | 需可控来源 |
| 幻觉率 | 高（30-70%） | 低（<5%） | 零容忍 |
| 引用可靠性 | 不确定 | 100% 可溯源 | 必须准确 |
| 用户控制 | 低（黑盒） | 高（全程可见） | 人在回路 |

**决策理由**：
学术写作者通常已读了大量论文，问题不是"信息不够"，而是"信息太多，组织不起来"。

**面试话术**：
> "我选择 Material Assembly 而非 Deep Research，是基于对用户场景的分析。学术写作者的痛点不是找不到信息，而是组织不起来。Deep Research 的幻觉率在学术场景是不可接受的，而 Material Assembly 能保证引用 100% 可追溯。"

---

### 2.3 决策三：写完 v0.2 后决定不采用

**决策**：v0.2 代码完成后决定不使用，直接进入 v1.0

**背景**：
原计划的版本演进是：v0.1 → v0.2 → v1.0
- v0.1：最小原型，验证 RAGFlow 连通性
- v0.2：基础流程，Query 生成优化、引用格式化（代码已完成，约 600 行）
- v1.0：Agent 链，专业化 Agent 编排

**分析过程**：

v0.1 测试后暴露的问题：
1. 单一 Prompt 难以处理复杂的多步骤任务
2. 检索质量依赖单一查询
3. 无法对检索结果进行排序筛选
4. 生成内容无法验证

这些问题需要的解决方案：

| v0.1 问题 | v0.2 方案 | v1.0 方案 |
|----------|----------|----------|
| 单一 Prompt 局限 | 更好的 Prompt | 专业化 Agent |
| 检索依赖单一查询 | Query 生成函数 | Query Generator Agent |
| 无法排序筛选 | 后处理逻辑 | Chunk Ranker Agent |
| 无法验证内容 | 无 | Verifier Agent |

**决策理由**：
1. **功能重叠**：v0.2 的 Query 生成、引用格式化在 v1.0 已以 Agent 形式实现
2. **增量价值不足**：v0.2 是"渐进式优化"，v1.0 是"架构升级"
3. **敢于放弃沉没成本**：v0.2 代码已写完（约 600 行），但识别到继续使用会产生技术债
4. **问题驱动**：v0.1 的问题需要 Agent 级别的解决方案

**类比解释**：
> "这就像装修房子。v0.1 是毛坯房，v0.2 是刷漆买家具，v1.0 是装智能家居系统。我已经买了家具（写完了 v0.2 代码），但发现最终目标是智能家居，那这些家具装智能系统时都要拆掉。与其继续用，不如现在就放弃。"

**面试话术**：
> "我写完了 v0.2 的代码（约 600 行），但在测试 v0.1 后发现，v0.2 是在试图把马车跑得更快，而 v1.0 是造汽车。v0.1 暴露的问题——检索不准、无法验证——需要的是架构升级，而不是更好的 Prompt。虽然 v0.2 代码已经写完，但我决定不使用它，因为继续用会产生技术债。这个决策体现了敢于放弃沉没成本的能力。"

## 3. 问题解决案例

### 3.1 案例一：RAGFlow 返回字段名不匹配

**问题现象**：
检索结果中 document_name 显示为 "Unknown"，无法正确显示文档名称。

**排查过程**：
1. 检查 RAGFlow API 文档 → 文档说返回 `document_name` 和 `score`
2. 打印实际返回数据 → 发现实际是 `document_keyword` 和 `similarity`
3. 确认是 API 版本差异导致

**解决方案**：
```python
# 兼容两种字段名
doc_name = chunk.get("document_keyword") or chunk.get("document_name", "Unknown source")
score = chunk.get("similarity") or chunk.get("score", 0)
```

**经验教训**：
调用第三方 API 时，应先调试确认实际返回的数据结构，而非依赖文档假设。

**面试话术**：
> "这个问题教会我一个原则：Trust but verify。API 文档可能过时或有版本差异，实际开发时要先打印真实返回数据确认。"

---

### 3.2 案例二：引用编号超出范围（引用幻觉）

**问题现象**：
生成的文本中出现 [28] 等引用编号，但实际只检索到 20 个 chunks。

**原因分析**：
1. LLM 在生成时可能"编造"不存在的引用编号
2. Prompt 中未明确限制引用编号范围

**解决方案**：
1. 修改 Prompt，明确限制：`只能使用 [1] 到 [{num_chunks}] 的编号`
2. 将实际检索到的 chunks 数量传递给生成函数
3. 在输出时只显示被引用的 chunks（解析生成文本中的 `[数字]` 模式）

```python
# Prompt 中明确约束
prompt = f"""...
2. 使用 [1], [2] 等标记引用，**只能使用 [1] 到 [{num_chunks}] 的编号**
..."""

# 后处理验证
cited_nums = set(int(m) for m in re.findall(r'\[(\d+)\]', draft))
for num in sorted(cited_nums):
    if 1 <= num <= len(chunks):  # 只显示有效引用
        # ...
```

**经验教训**：
LLM 生成引用时容易产生幻觉，必须在 Prompt 中明确约束，并在后处理中验证。

**面试话术**：
> "这是一个典型的 LLM 幻觉问题。解决方案是双重保险：Prompt 层面明确约束范围，代码层面做后处理验证。这个经验后来也影响了 v1.0 的设计——我专门设计了 Verifier Agent 来做引用验证。"

---

### 3.3 案例三：输出语言不匹配

**问题现象**：
用户输入中文主题，但输出为英文段落。

**原因分析**：
原始 Prompt 使用英文编写，未指定输出语言。LLM 倾向于用 Prompt 的语言输出。

**解决方案**：
修改 Prompt 为中文，并明确要求：
- 输出语言为中文
- 专业术语保留英文（如 Deep Research, RAG, LLM 等）

**经验教训**：
Prompt 的语言和输出要求需要明确指定，不能依赖 LLM 自动推断。

---

### 3.4 案例四：RAGFlow API 连接超时

**问题现象**：
Gradio UI 中点击生成后报错：`httpx.ConnectTimeout`

**解决方案**：
1. 将超时时间从 30s 增加到 60s
2. 添加更友好的错误提示

```python
async with httpx.AsyncClient(timeout=60.0) as client:
    try:
        response = await client.post(url, json=payload, headers=headers)
    except httpx.ConnectTimeout:
        raise Exception("RAGFlow API 连接超时，请检查网络或稍后重试")
```

**经验教训**：
网络请求需要合理的超时设置和友好的错误处理。

> "v1.0 采用 Chain-of-Agents 架构，每个 Agent 职责单一。比如 Query Generator 专门负责把用户输入扩展成多个检索查询，Verifier 专门负责验证引用是否有原文支撑。这种设计的好处是可独立测试、易于调试。"

## 5. 预期面试问答

### Q1: 为什么选择这个项目？

> "我在准备毕业论文时发现一个痛点：已经读了很多文献，但组织成连贯的段落很耗时。市面上的 AI 写作工具要么幻觉严重，要么引用不可追溯。我想解决这个问题，同时也想通过这个项目展示从 0 到 1 的架构演进能力。"

### Q2: 项目最大的技术挑战是什么？

> "最大的挑战是控制 LLM 的引用幻觉。LLM 很容易编造不存在的引用编号。我的解决方案是双重保险：Prompt 层面明确约束范围，代码层面做后处理验证。这个经验也影响了后续 Agent 架构的设计——我专门设计了 Verifier Agent。"

### Q3: 为什么写完 v0.2 后决定不用？

> "我写完了 v0.2 的代码（约 600 行），但在测试 v0.1 后发现，v0.2 是在试图把马车跑得更快，而 v1.0 是造汽车。v0.1 暴露的问题——检索不准、无法验证——需要的是架构升级，而非 Prompt 优化。虽然代码已经写完，但我决定不使用它。这个决策体现了敢于放弃沉没成本的能力——识别出不应该继续使用的代码，比一开始就不写更难。"

### Q4: 为什么用 RAGFlow 而不是自建 RAG？

> "PDF 解析是'已解决的问题'，不是我的核心价值。RAGFlow 在表格、公式、图片解析上已经投入了大量工程优化，我无法超越也没必要重复。这让我能把精力放在真正的创新点——写作流程的 Agent 编排上。"

### Q5: 如果重新做这个项目，会有什么不同？

> "我会更早地建立评测体系。目前的问题是缺乏量化数据来证明 v1.0 比 v0.1 好多少。如果有 20-50 个测试用例的对比数据，跳过 v0.2 的决策会更有说服力。"

### Q6: Agent 架构相比单一 Prompt 的优势是什么？

> "三个优势：
> 1. **可调试性**：每个 Agent 可以独立测试，出问题容易定位
> 2. **可扩展性**：新增功能只需添加新 Agent，不影响现有逻辑
> 3. **质量控制**：可以在关键节点（如 Verifier）做质量把关
>
> 当然也有代价：延迟增加、Token 消耗增加。这是一个权衡。"

### Q7: 这个项目的局限性是什么？

> "主要有三个局限：
> 1. **响应时间长**：端到端 20-40 秒，用户体验不佳
> 2. **依赖文献质量**：如果用户上传的文献不相关，生成质量会下降
> 3. **缺乏评测数据**：目前没有量化的准确率指标
>
> 这些是后续版本需要解决的问题。"

### Q8: 你从这个项目学到了什么？

> "三个主要收获：
> 1. **架构决策要基于真实数据**：不是臆测哪个方案更好，而是先跑通再决定
> 2. **识别核心价值**：PDF 解析不是我的价值，写作流程才是
> 3. **敢于跳过不必要的步骤**：如果问题需要架构升级，就不要在中间版本浪费时间"

### 项目一句话介绍
> ManuScript 是一个基于 RAG 的学术写作辅助工具，帮助用户基于自己的文献库生成带准确引用的论文段落。

### 核心技术栈
- Python 3.10+ / Gradio / FastAPI
- RAGFlow（RAG 引擎）/ OpenAI API（LLM）
- LangGraph（Agent 编排）

### 三个关键决策
1. RAGFlow 托管 vs 自建 RAG → 选择托管，聚焦核心价值
2. Deep Research vs Material Assembly → 选择 Material Assembly，保证引用可追溯
3. v0.2 代码完成后不采用 → 敢于放弃沉没成本，问题需要架构升级

### 三个解决的问题
1. 引用幻觉 → Prompt 约束 + 后处理验证
2. API 字段不匹配 → 兼容性处理
3. 输出语言不匹配 → 明确指定输出要求
