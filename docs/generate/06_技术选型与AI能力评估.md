## 2. RAG 技术评估

### 2.1 RAG 原理简述

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        RAG 工作原理                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   传统 LLM：                                                            │
│   ┌─────────────┐                    ┌─────────────┐                   │
│   │  用户问题   │ ──────────────────► │  LLM 回答   │                   │
│   └─────────────┘                    └─────────────┘                   │
│   问题：LLM 只能基于训练数据，容易产生幻觉                              │
│                                                                         │
│   RAG 增强：                                                            │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐               │
│   │  用户问题   │ ──►│  向量检索   │ ──►│  相关文档   │               │
│   └─────────────┘    └─────────────┘    └──────┬──────┘               │
│                                                │                       │
│                                                ▼                       │
│   ┌─────────────┐    ┌─────────────────────────────────┐              │
│   │  LLM 回答   │ ◄──│  问题 + 相关文档 (Context)      │              │
│   └─────────────┘    └─────────────────────────────────┘              │
│   优势：LLM 基于检索到的真实文档回答，减少幻觉                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 RAG 技术挑战

| 环节 | 挑战 | 影响 | ManuScript 应对 |
|------|------|------|----------------|
| **PDF 解析** | 表格错乱、公式乱码、双栏混排 | 检索质量下降 | RAGFlow 专业解析 |
| **文本切分** | 语义边界难以把握 | 信息碎片化 | RAGFlow 智能切分 |
| **向量化** | 通用 Embedding 学术适配差 | 相关性不准 | RAGFlow 多模态 Embedding |
| **检索** | 语义漂移、相关性不足 | 召回质量差 | 多路检索 + 重排序 |
| **生成** | 幻觉、引用不准 | 用户不信任 | Verifier + 强制引用 |

### 2.3 自建 RAG vs RAGFlow 托管

#### 决策分析

| 维度 | 自建 RAG | RAGFlow 托管 | 选择 |
|------|---------|-------------|------|
| **PDF 解析质量** | 60-70%（需大量优化） | 90%+（专业团队优化） | RAGFlow ✅ |
| **工程投入** | 高（3-6个月） | 低（API 对接） | RAGFlow ✅ |
| **维护成本** | 持续投入 | 几乎无 | RAGFlow ✅ |
| **灵活性** | 完全可控 | 受限于 API | 自建 ✅ |
| **学术场景适配** | 需自行优化 | 已有优化 | RAGFlow ✅ |

#### 决策结论

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     RAG 选型决策                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【决策】使用 RAGFlow 托管                                              │
│                                                                         │
│  【理由】                                                               │
│  1. PDF 解析是"已解决的问题"，不是 ManuScript 的核心价值               │
│  2. RAGFlow 已投入大量工程优化，我们无法超越                            │
│  3. 节省 60% 工程量，聚焦写作流程创新                                   │
│                                                                         │
│  【风险】                                                               │
│  • API 可用性依赖外部服务                                               │
│  • 定制能力受限                                                         │
│                                                                         │
│  【缓解】                                                               │
│  • 实现重试和降级机制                                                   │
│  • 抽象 RAG 接口，支持未来替换                                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.4 RAGFlow 能力评估

| 能力 | 评估 | 说明 |
|------|------|------|
| **PDF 解析** | ★★★★★ | 支持复杂排版、表格、公式 |
| **多模态** | ★★★★☆ | 支持图片描述提取 |
| **向量检索** | ★★★★☆ | 支持多种 Embedding |
| **API 稳定性** | ★★★★☆ | 正式服务，有 SLA |
| **成本** | ★★★☆☆ | 按量付费，需控制 |

## 3. LLM 能力与局限

### 3.1 LLM 在学术写作中的能力边界

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     LLM 能力边界分析                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   【LLM 擅长】                                                          │
│   ✅ 语言组织和润色                                                     │
│   ✅ 结构化信息整合                                                     │
│   ✅ 风格转换和改写                                                     │
│   ✅ 基于给定内容的总结                                                 │
│   ✅ 格式转换（如生成引用格式）                                         │
│                                                                         │
│   【LLM 不擅长】                                                        │
│   ❌ 确保事实准确性（容易幻觉）                                         │
│   ❌ 精确引用（会编造文献）                                             │
│   ❌ 数字和数据的准确性                                                 │
│   ❌ 长文本的一致性维护                                                 │
│   ❌ 领域专业知识的准确性                                               │
│                                                                         │
│   【ManuScript 的设计原则】                                             │
│   用 LLM 做它擅长的事（组织语言），                                     │
│   用 RAG 和流程设计弥补它的不足（确保准确性）                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 LLM 幻觉问题深度分析

#### 幻觉类型分类

| 幻觉类型 | 描述 | 学术写作中的表现 | 危害程度 |
|---------|------|-----------------|---------|
| **事实幻觉** | 编造不存在的事实 | "研究表明 X 导致 Y" | 极高 |
| **引用幻觉** | 编造不存在的文献 | "[Smith 2020] 指出..." | 极高 |
| **数据幻觉** | 编造数字和数据 | "准确率达到 96.4%" | 极高 |
| **逻辑幻觉** | 不正确的推理 | 因果关系错误 | 高 |
| **一致性幻觉** | 前后矛盾 | 不同段落说法不一 | 中 |

#### 幻觉率统计（基于研究数据）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     LLM 幻觉率统计                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   【开放域生成（无 RAG）】                                              │
│   • 事实性陈述幻觉率：30-50%                                            │
│   • 引用幻觉率：70-90%（几乎必然编造）                                  │
│   • 数据幻觉率：50-70%                                                  │
│                                                                         │
│   【RAG 增强后】                                                        │
│   • 事实性陈述幻觉率：5-15%（大幅下降）                                 │
│   • 引用幻觉率：取决于 Prompt 设计                                      │
│   • 数据幻觉率：10-20%                                                  │
│                                                                         │
│   【封闭域 + 强制引用】                                                 │
│   • 引用幻觉率：<1%（从源头杜绝）                                       │
│   • 内容幻觉率：<5%（Verifier 进一步降低）                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 LLM 选型对比

| 模型 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **GPT-4** | 综合能力最强，指令遵循好 | 成本高，速度慢 | 复杂任务（Planner, Writer） |
| **GPT-4o** | 速度快，成本适中 | 复杂推理稍弱 | 大部分任务 |
| **GPT-3.5** | 成本低，速度快 | 指令遵循差，幻觉多 | 简单任务（格式化） |
| **Claude 3** | 长文本能力强，幻觉较少 | API 限制 | 长文档处理 |
| **国产模型** | 成本低，中文好 | 英文学术能力弱 | 中文场景备选 |

## 4. Agent 架构选型

### 4.1 Agent 架构模式对比

#### 模式一：单一 Prompt（最简单）

```
用户输入 ──────────────────────────────────────► LLM ──► 输出

优点：简单
缺点：不可控，难以分步调试
适用：简单任务
```

#### 模式二：Prompt Chain（链式）

```
用户输入 ──► Prompt 1 ──► Prompt 2 ──► Prompt 3 ──► 输出

优点：可分步调试
缺点：缺乏专业化
适用：流程固定的任务
```

#### 模式三：Chain-of-Agents（专业化链）

```
用户输入 ──► Agent 1 ──► Agent 2 ──► Agent 3 ──► 输出
            (专业1)     (专业2)     (专业3)

优点：专业化分工，可独立优化
缺点：架构复杂度增加
适用：需要专业分工的任务
```

#### 模式四：Orchestrator-Worker（动态调度）

```
                    ┌─────────────┐
                    │ Orchestrator│
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
      ┌─────────┐    ┌─────────┐    ┌─────────┐
      │ Worker 1│    │ Worker 2│    │ Worker N│
      └─────────┘    └─────────┘    └─────────┘

优点：高度灵活，支持并行
缺点：架构最复杂
适用：任务复杂度动态变化
```

## 5. 幻觉控制策略

### 5.1 多层防御体系

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     幻觉控制多层防御                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Layer 1: 封闭域限制                                                   │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │  只使用用户上传的文献库，从源头杜绝"编造文献"的可能              │  │
│   │  效果：引用幻觉率从 70-90% 降到 <5%                              │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                               │                                         │
│                               ▼                                         │
│   Layer 2: 强制引用标注                                                 │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │  Prompt 明确要求：每句话必须标注 [Doc_ID:Page]                   │  │
│   │  效果：迫使 LLM 将内容锚定在具体来源上                           │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                               │                                         │
│                               ▼                                         │
│   Layer 3: Verifier Agent                                               │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │  独立 Agent 逐句检查草稿是否有原文支撑                           │  │
│   │  效果：识别并标记潜在幻觉                                        │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                               │                                         │
│                               ▼                                         │
│   Layer 4: 用户确认                                                     │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │  用户先确认素材，再生成草稿；可点击引用查看原文                  │  │
│   │  效果：人在回路，最后一道防线                                    │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│   【综合效果】                                                          │

### 6.2 关键技术决策记录

#### 决策 1：Frontend 选型

```
【问题】MVP 阶段如何快速搭建 UI？

【选项】
A. Gradio - Python 原生，组件丰富，快速
B. Streamlit - Python 原生，交互强
C. React - 生产级，需要前端开发

【决策】Gradio (MVP) → React (Production)

【理由】
• MVP 阶段用 Gradio 快速验证，几小时搭建完整 UI
• 产品验证后再投入 React 开发
• 渐进式迁移，降低风险
```

#### 决策 2：Agent 框架选型

```
【问题】如何编排多个 Agent？

【选项】
A. LangChain - 生态大，但抽象层过多
B. LangGraph - 工作流清晰，状态管理好
C. 自建 - 完全可控，但需要更多开发

【决策】LangGraph (v1.0) / 自建 (v0.x)

【理由】
• v0.1/v0.2 简单场景自建即可
• v1.0 引入 LangGraph 管理复杂工作流
• LangGraph 的状态机模型适合我们的流程
```

#### 决策 3：存储选型

```
【问题】如何存储项目/大纲/草稿数据？

【选项】
A. SQLite - 轻量，无需部署
B. PostgreSQL - 功能强大，生产级
C. MongoDB - 文档型，灵活

【决策】SQLite (MVP) → PostgreSQL (Production)

【理由】

### 7.2 风险缓解方案

#### RAGFlow 可用性风险

```python
# RAGFlow 调用重试机制

from tenacity import retry, stop_after_attempt, wait_exponential

class RAGFlowClient:
    @retry(
        stop=stop_after_attempt(3),
        wait=wait_exponential(multiplier=1, min=1, max=10)
    )
    async def search(self, query: str) -> List[Chunk]:
        try:
            response = await self._call_api(query)
            return response
        except Exception as e:
            logger.error(f"RAGFlow API error: {e}")
            raise

    async def search_with_fallback(self, query: str) -> List[Chunk]:
        try:
            return await self.search(query)
        except Exception:
            # 降级：返回缓存结果或空列表
            return self._get_cached_results(query) or []
```

#### LLM 成本控制

```python
# Token 使用监控

class TokenBudget:
    def __init__(self, daily_limit: int = 100000):
        self.daily_limit = daily_limit
        self.used_today = 0

    def can_proceed(self, estimated_tokens: int) -> bool:
        return self.used_today + estimated_tokens < self.daily_limit

    def record_usage(self, tokens: int):
        self.used_today += tokens
        if self.used_today > self.daily_limit * 0.8:
            logger.warning(f"Token usage at {self.used_today}/{self.daily_limit}")
```
