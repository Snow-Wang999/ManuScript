## 1. PDF解析方案选择

### 选项对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **RAGFlow** | 开箱即用、内置向量化、支持多种格式 | 重量级、定制性差、依赖其服务 | 快速搭建原型 |
| **传统方法** (PyMuPDF/pdfplumber) | 轻量、可控、本地化好 | 复杂布局处理差、需自己处理表格 | 简单PDF |

| **Docling** (IBM) | 开源、表格处理强、本地运行 | 较新、社区小 | 复杂表格PDF |
| **Unstructured.io** | 功能全面 | 商业版才效果好 | 企业级 |

### 我的建议
对于学术手稿项目，推荐 **Marker + PyMuPDF 组合**：
- Marker 处理复杂学术PDF（公式、图表、引用）
- PyMuPDF 作为fallback处理简单PDF
- 完全本地化，无需外部API

## 2. 本地化与数据脱敏

### 完全本地化可行吗？
**可以，但有trade-off：**

```
本地化方案架构：
┌─────────────────────────────────────┐
│           本地处理层                 │
├──────────┬──────────┬───────────────┤
│ PDF解析  │ 向量化   │  LLM推理      │
│ Marker   │ BGE-M3   │  Ollama/vLLM  │
│ PyMuPDF  │ (本地)   │  (本地)       │
└──────────┴──────────┴───────────────┘
```

| 组件 | 本地方案 | 硬件需求 |
|------|----------|----------|
| Embedding | BGE-M3 / bge-large-zh | 4GB显存够用 |
| LLM | Qwen2.5-7B / Llama3-8B | 16GB显存 |
| 向量DB | Milvus Lite / Chroma | 纯CPU |

### 脱敏处理
如果数据敏感，可在**解析后、存储前**加脱敏层：
```python
# 伪代码
parsed_text = marker.parse(pdf)
anonymized = anonymizer.process(parsed_text)  # 脱敏
vectors = embedder.encode(anonymized)  # 向量化
```

## 3. LaTeX解析为什么复杂？

LaTeX复杂的原因：

1. **宏展开问题**：`\newcommand` 定义的宏需要递归展开
2. **环境嵌套**：`\begin{theorem}` 内嵌 `\begin{equation}` 等
3. **数学公式多形态**：`$...$`, `$$...$$`, `\[...\]`, `equation`, `align`...
4. **交叉引用**：`\ref`, `\cite` 需要多遍编译才能解析
5. **包依赖**：不同的包改变命令行为
6. **非标准语法**：作者自定义命令

### 解决方案
- **如果只要文本**：正则 + pylatexenc 够用
- **如果要结构化**：TexSoup 或 plasTeX
- **如果要渲染公式**：KaTeX/MathJax 转HTML/SVG

## 4. 向量数据库 vs Markdown + grep

这是**核心架构决策**，我的分析：

| 维度 | 向量数据库 | Markdown + grep |
|------|------------|-----------------|
| **语义搜索** | ✅ 强 | ❌ 只有关键词 |
| **复杂度** | 高（需维护） | 低（文件系统） |
| **成本** | Embedding计算成本 | 几乎为零 |
| **精确匹配** | 一般 | ✅ 强 |
| **可调试性** | 黑盒 | ✅ 透明 |

### 我的建议：**混合方案**

```
┌────────────────────────────────────────┐
│              检索层                     │
├──────────────────┬─────────────────────┤
│   精确搜索       │    语义搜索          │
│   ripgrep        │    向量数据库        │
│   (关键词/正则)   │    (相似性)         │
└────────────────────────────────────────┘
        ↓                    ↓
        └────── Reranker ────┘
                   ↓
             合并排序结果
```

**具体做法：**
1. **所有文档存Markdown**（可grep、可版本控制、可读）
2. **同时建向量索引**（支持语义搜索）
3. 检索时**双路召回** + Reranker重排

## 5. RAG检索怎么做？

### 基础RAG流程
```
用户问题 → Embedding → 向量检索 Top-K → 拼接Prompt → LLM生成
```

### 学术场景优化的RAG
```
┌─────────────────────────────────────────────────────────┐
│                   Query Processing                       │
│  用户问题 → 意图识别 → Query重写/扩展                     │
└───────────────────────┬─────────────────────────────────┘
                        ↓
┌───────────────────────┴─────────────────────────────────┐
│                   Hybrid Retrieval                       │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────┐  │
│  │ 稠密向量    │  │ 稀疏向量     │  │ 关键词BM25     │  │
│  │ (BGE-M3)   │  │ (SPLADE)    │  │ (Elasticsearch)│  │
│  └─────────────┘  └──────────────┘  └────────────────┘  │
└───────────────────────┬─────────────────────────────────┘
                        ↓
┌───────────────────────┴─────────────────────────────────┐
│                   Reranking                              │
│  粗排结果 → bge-reranker-v2 → 精排Top-N                  │
└───────────────────────┬─────────────────────────────────┘
                        ↓
┌───────────────────────┴─────────────────────────────────┐
│                   Context Assembly                       │
│  结构化组装：元数据 + 章节上下文 + 相关chunk             │
└───────────────────────┬─────────────────────────────────┘
                        ↓
                      LLM生成
```
