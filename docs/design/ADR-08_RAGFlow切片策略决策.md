## 待决策事项（v1.0 跑通后再评估）

### 1. Metadata 过滤

**决策点**：是否需要在检索时引入 metadata 过滤？

**触发条件**（出现以下问题时再引入）：
- 检索结果混入不相关领域的文献
- 用户反馈"引用的文献太旧了"
- 需要支持多个独立的文献库
- 文献库规模 > 100 篇且跨多个研究领域

**可选过滤维度**：
- `publish_year` - 发表年份
- `doc_type` - 文献类型（journal/conference/preprint）
- `domain` - 研究领域
- `project_id` - 项目隔离

**当前决策**：暂不实现，v1.0 使用纯语义检索验证核心流程。

---

### 2. VLM 模型升级（表格解析）

**决策点**：是否需要升级 RAGFlow 的 VLM 模型（当前 qwen-vl-plus）？

**当前决策**：暂不升级。表格内容通常不是引用重点，优先在 Chunk Ranker 中过滤低质量表格 chunk。

---

### 3. 后检索重切片（Post-retrieval Re-chunking）

**决策点**：是否需要在检索后对 chunk 进行二次处理？

**两种方案**：
- **上下文扩展**：获取 chunk 前后文，拼接成更完整的段落
- **重新切片**：基于语义边界重新划分 chunk

**当前决策**：先测试 v1.0 的 Chunk Ranker 效果，如果检索质量仍有问题再考虑实现。

---

### 4. 父子检索（Parent-Child Retrieval）

**决策点**：是否需要实现父子检索策略？

**什么是父子检索**：
- 小 chunk 用于检索（精确匹配）
- 父 chunk 用于生成（完整上下文）

**触发条件**（出现以下问题时再考虑）：
- LLM 生成时频繁"断章取义"
- 引用的上下文不足以支撑论点
- 检索到的 chunk 太碎片化

**实现成本**：
- RAGFlow 不原生支持，需要自建索引
- 需要维护 chunk 层级关系
- 自定义切片策略

**替代方案**（更轻量）：
- 上下文扩展：获取 chunk 前后各 500 字符
- 增大 chunk_token_num：在 RAGFlow 配置中设置 1024-2048

**当前决策**：暂不实现。等 v1.0 跑通后，如果发现上下文不足问题，先尝试"上下文扩展"，成本更低。

## 补充资料：RAGFlow 混合检索机制

### BM25 + 向量检索（自动生效）

RAGFlow 内部默认使用混合检索，**无需在代码中显式调用**：

```
你的代码 (prototype.py)          RAGFlow 内部 (自动)
        │                              │
        ▼                              ▼
payload = {                    ┌─────────────────┐
  "question": query,           │  全文检索 (BM25) │ ← 5%
  "dataset_ids": [...],        └────────┬────────┘
  "top_k": 5                            │
}                              ┌────────┴────────┐
        │                      │  Fusion 融合     │
        ▼                      │  weights: 0.05, 0.95
POST /api/v1/retrieval         └────────┬────────┘
                               ┌────────┴────────┐
                               │  向量检索 (cosine)│ ← 95%
                               └─────────────────┘
```

### 默认权重配置

来源：RAGFlow `search.py:127`

```python
fusionExpr = FusionExpr("weighted_sum", topk, {"weights": "0.05,0.95"})
# 5% 全文检索 (BM25) + 95% 向量检索
```

### 什么时候需要调整 BM25 权重

| 场景 | 建议 |
|------|------|
| 学术术语/专有名词检索不准 | 提高 BM25 权重（专有名词靠关键词匹配更准） |
| 语义理解不够 | 保持当前配置（95% 向量检索） |
| 想自定义权重 | 在 API 调用中加 `vector_similarity_weight` 参数 |

### 当前决策

暂不调整，RAGFlow 默认配置已启用混合检索。等 v1.0 跑通后，如果发现专有名词检索不准，再考虑调整权重。

## 备选方案：PDF → 图片 → VLM → Markdown

### 方案概述

绕过传统 PDF 解析，直接用视觉模型"看"页面：

```
PDF 页面
    │
    ▼ (渲染为图片)
┌─────────────────┐
│  PNG/JPG 图片    │
└────────┬────────┘
         │
         ▼ (VLM 识别)
┌─────────────────┐
│  GPT-4V / Claude │  ← 视觉模型"看"整个页面
│  Qwen-VL / etc   │
└────────┬────────┘
         │
         ▼
    Markdown 文本
```

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **直接解析 PDF** | 快、便宜、文本精确 | 布局可能丢失、表格/公式差 |
| **PDF → 图片 → VLM** | 布局保留好、表格/公式识别强 | 慢、贵、可能有 OCR 误差 |

### 效果评估

| 内容类型 | 效果 |
|---------|------|
| 纯文本 | ✅ 好（但可能有 OCR 小错误） |
| 表格 | ✅ 很好（VLM 擅长识别表格结构） |
| 公式 | ✅ 好（可转 LaTeX） |
| 图表 | ⚠️ 可描述，但无法还原原图 |
| 多栏布局 | ✅ 好（VLM 能理解布局） |

### 成本估算

```
假设：100 页 PDF

直接解析（RAGFlow DeepDoc）:
  - 成本: ~$0（本地模型）
  - 时间: ~1-2 分钟

图片 + VLM（GPT-4V）:
  - 成本: ~$1-3（每页 $0.01-0.03）
  - 时间: ~5-10 分钟
```

### 相关工具

| 工具 | 方案 |
|------|------|
| **Nougat** (Meta) | PDF → 图片 → 专用模型 → Markdown |
| **Marker** | 混合：先尝试直接解析，复杂页用 VLM |
| **RAGFlow VisionParser** | PDF → 图片 → VLM 描述 |

### 适用场景

**适合**：
- 学术论文（表格、公式多）
- 扫描版 PDF
- 对布局要求高

**不适合**：
- 大批量处理（成本高）
- 纯文本文档（杀鸡用牛刀）

### 当前决策

暂不使用。当前 RAGFlow DeepDoc 的解析质量足够 MVP 验证。如果后续发现表格/公式解析是核心痛点，再考虑此方案。
