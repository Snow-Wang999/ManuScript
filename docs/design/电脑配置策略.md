## 资源消耗分析

### 1. PDF解析（Marker）

| 指标 | 无GPU | 有GPU（4GB+） |
|------|-------|---------------|
| 速度 | ~30秒/篇 | ~5秒/篇 |
| 内存 | 2-4GB | 2-4GB |
| CPU | 高占用 | 低占用 |

**结论**：Marker 用 CPU 也能跑，只是慢一些。对于一次性导入50-200篇的场景，**可接受**。

---

### 2. Embedding 向量化

这是**资源消耗的关键瓶颈**：

| 模式 | 模型 | 显存需求 | 速度 | 成本 |
|------|------|----------|------|------|
| **本地 GPU** | BGE-M3 | 2-4GB | ~100段/秒 | 免费 |
| **本地 CPU** | BGE-M3 | 0（用内存8GB） | ~5段/秒 | 免费但极慢 |
| **API** | OpenAI embedding | 0 | ~50段/秒 | ~$0.02/百万token |

**实际场景估算**（200篇论文，每篇约100个段落）：

```
总段落数：200 × 100 = 20,000 段
平均每段：~200 tokens

本地GPU：20,000 ÷ 100 = 200秒 ≈ 3分钟
本地CPU：20,000 ÷ 5 = 4000秒 ≈ 1小时+
API调用：20,000 ÷ 50 = 400秒 ≈ 7分钟，成本约 $0.08
```

---

### 4. 全文检索（ripgrep + FTS5）

## 🎯 我的建议

针对你的"本地优先"定位，建议采用**渐进式策略**：

```
┌─────────────────────────────────────────────────────────────┐
│                    首次导入（重）                            │
├─────────────────────────────────────────────────────────────┤
│  检测硬件 → 有GPU？                                          │
│     ├─ 是 → 本地Marker + 本地BGE-M3                         │
│     └─ 否 → 本地Marker(CPU) + 提示用户选择：                 │
│              ├─ "本地处理（约1小时）"                        │
│              └─ "API加速（需联网，约$0.1）"                  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    日常检索（轻）                            │
├─────────────────────────────────────────────────────────────┤
│  ripgrep + FTS5 + ChromaDB                                  │
│  → 任何设备都能秒级响应，无硬件依赖                           │
└─────────────────────────────────────────────────────────────┘
```

**关键设计原则**：
1. **重活只做一次**：解析和向量化只在导入时执行
2. **轻活随时做**：检索几乎无消耗
3. **给用户选择权**：无GPU时让用户决定等待还是用API
