1) 用户旅程重新定义是否按原文那样？
原文“探索→结构化→写作→润色”的四阶段是合理的，但我建议把它理解为“可回退的状态机”，而不是线性流程。现实里用户会反复“探索↔结构化”、“写作↔探索”。所以更合适的表述是：

系统需要支持“跨阶段跳转”和“局部再探索”。
这点是原文没有强调的。
2) 架构分层中 Hook 怎么用？
Hook 本质是“可插拔的预处理/后处理点”，常见放置位置：

输入前置：脱敏/去隐私 → 再进入检索或云处理
检索后置：结果过滤/格式规范化
写作后置：引用校验、格式转换、导出
它不是一个“新层”，而是插在流程关键节点的“策略点”。如果当成独立层，会造成职责混乱。
建议：把 Hook 当“管道插槽”，由主 Agent 或路由层决定是否启用。
3) 分层模型是否清晰？
整体方向清晰，但存在两个模糊点：

路由层 vs 主 Agent：谁做意图识别，谁做任务编排？需要定边界。
Sub-agent vs MCP 工具：是“能力封装”还是“执行者”？如果 Sub-agent 只是调用工具，那很多 Sub-agent 可以退化成工具调用。
结论：模型清晰但仍“概念化”，缺少“输入/输出契约”。
4) 为什么对话历史是短期记忆？
这里有误解。通常“短期记忆”指的是：

容量有限（上下文窗口），
只保留与当前任务强相关的片段，
会被压缩/替换。
对话历史可以“物理上存在”，但它在推理时通常以“压缩摘要+关键片段”形式进入模型，所以称为短期。
长期记忆则是：用户偏好、论文项目进度、已确认的结论等，独立存储，可跨会话复用。
所以“对话历史短期”强调的是“推理可用性”，不是“存储寿命”。
5) 是否需要 Markdown 存储？
不是必须，但有明显优势：

可追溯、可读、易迁移、可直接检索
与“用户可编辑笔记”匹配
但它带来工程成本（解析、同步、索引维护）。
决策点：
你是否把“可追溯原文”当刚需？如果是，Markdown 很有价值。
如果你更重视“语义召回准确度”，向量库更优，但必须建立引用定位机制。
6) 向量库与 Markdown 的关系
混合方案通常是：

Markdown 是“源文档”，
向量库只存“切分后的片段 + 元数据 + 位置信息”。
检索返回片段后，按元数据定位回 Markdown 段落。
7) 笔记修改后，向量库要不要更新？
要看你是否把“用户笔记”纳入向量检索：

如果向量库只索引论文正文，笔记变了不用动。
如果需要“检索笔记内容”，就必须更新对应索引。
因此要明确：向量索引范围是否包含笔记区。
8) 混合方案检索难度大吗？如何决策？
难度主要在两点：

原文更新 → 向量索引同步
检索命中 → 原文定位
怎么选：
轻量版本：Markdown 全文检索 + 少量结构化标签（不做向量）
进阶版本：Markdown 为源 + 向量库索引正文
复杂版本：Markdown + 向量库 + 引用定位 + 多版本笔记
建议按“你最关键的用户价值”来定：
如果“追溯原文”是第一优先 → Markdown 必要
如果“检索质量”是第一优先 → 向量库优先
9) 多轮对话与并行对话：共享记忆怎么设计？
可以理解为三层共享：

全局共享：论文库、用户偏好、项目状态
会话共享：当前会话的临时上下文
引用共享：允许“引用其它会话摘要”
这就是原文 A/B/C 的更完整版本。
推荐：B + C 组合
会话独立，但允许引用摘要
全局项目状态可见

常见触发点：

会话超过上下文阈值时自动压缩
任务完成后生成“持久摘要”
