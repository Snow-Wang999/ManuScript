# ADR-12 v1.0 弃用与 v2.0 处理说明

**状态**: 已确认  
**日期**: 2026-02-04  
**关联**: `docs/design/ADR-11_ManuScript_v3.0_技术架构.md`
**参考**: `docs/generate/05_架构版本规划与实现路径.md`

---

## 背景
v1.0 与 v2.0 版本的设计目标与架构路线，已被 v3.0 “对话式检索 + 可追溯溯源 + 本地优先”的主线决策替代。为避免后续开发中出现路线回摆或混用，需要明确 v1.0 的弃用原因，以及 v2.0 的保留价值与使用边界。

---

## 决策

### 1) v1.0 视为弃用（Deprecated）
v1.0 的核心定位为 **“论文生成器 + 固定链式 Agent”**，与 v3.0 的 **“证据与溯源为主产出、生成文本作为解释层”** 的方向冲突。  
因此 v1.0 不再作为开发目标，仅保留为历史记录。

### 2) v2.0 作为“实验/对比基线”保留（Archived Baseline）
v2.0 作为 Orchestrator-Worker 的架构实验，保留用于：
- 说明“为何选择 v3.0 路线”（对比论证）
- 回溯技术尝试与失败点（风险备份）
- 复用少量可独立迁移的脚本/模块（若存在）

---

## 理由

## v1.0 / v2.0 架构概览（摘录）

### v1.0：Chain-of-Agents 静态链
- **设计理念**：专业化 Agent 分工，按固定链条处理。
- **流程**：Section Planner → Query Generator → Retrieval → Chunk Ranker → Draft Writer → Verifier。
- **特点**：串行链路、上下文在多个 Agent 间传递。
- **问题预期**：延迟累积、链路失败点多、上下文成本高。

### v2.0：Orchestrator-Worker 动态调度
- **设计理念**：由 Orchestrator 判断复杂度，按需调用不同 Worker。
- **流程**：复杂度分析 → 选择 Worker（simple/complex/review）→ 并行执行 → 汇总。
- **特点**：动态调度、支持并行、多策略处理。
- **风险预期**：复杂度判断不准、并行 Token 超限、调试复杂度增加。

---

## v2.0 相对 v1.0 的优势

1) **调度更灵活**：v1.0 固定 6 步链，所有章节同样处理；v2.0 可按复杂度选择不同路径，避免过度处理。  
2) **并行能力**：v2.0 支持多章节并行执行，适合批量处理（尤其是综述/多章节任务）。  
3) **容错更好**：v2.0 可局部重试或跳过，不会“一步错全链断”。  
4) **策略可演进**：v2.0 可逐步加入不同 Worker 策略（例如综述/方法/对比），比 v1.0 更容易扩展。

---

## v2.0 vs v3.0：Agent 设计差异（ADR 对比说明）

### 核心目标
- **v2.0**：面向“写作生成流程”，Agent 负责规划 → 检索 → 排序 → 生成 → 验证。  
- **v3.0**：面向“证据与溯源闭环”，Agent 主要做意图识别与工具路由，证据是主产出，生成文本作为解释层。

### 编排结构
- **v2.0**：Orchestrator-Worker 动态调度多 Worker，按章节复杂度分流。  
- **v3.0**：单主 Agent + 无状态工具（Search/Notes/Summarizer/Export），最短路径完成任务。

### 上下文与成本
- **v2.0**：多 Worker 反复读取上下文，Token 与时延成本较高。  
- **v3.0**：主 Agent 持有会话状态，工具即生即死，避免上下文重复加载。

### 可解释性与调试
- **v2.0**：执行路径多、分支多，回放与定位问题成本较高。  
- **v3.0**：工具调用链短、结果可直接溯源，解释成本更低。

### 相似点
- 都需要**意图/复杂度判断**（v2.0 复杂度判断，v3.0 意图路由）。  
- 都依赖检索，但 v2.0 把检索当中间环节，v3.0 把检索结果作为核心输出之一。

---

### v1.0 弃用的核心理由
- **产品定位偏移**：以“生成段落”为主产出不符合“证据与溯源优先”的目标。
- **架构复杂度过高**：多 Agent 固定链条降低可控性，面试演示解释成本高。
- **风险集中**：生成环节的幻觉风险与可解释性不足，削弱核心价值主张。
- **链式 Agent 的上下文成本过高**：链式流程需要多个 Agent 反复读取共享上下文，增加时延与 Token 成本；在 v3.0 这种“检索优先”的任务里，一个主 Agent + 工具路由反而更清晰、成本更低。

**给面试官的解释（可直接复述）**  
v1.0 的核心能力是“多 Agent 产出完整论文草稿”，但在真实场景里最容易被质疑的不是写得快，而是**是否可验证与可追溯**。  
v3.0 的主线把“每一句话都有原文出处”作为首要目标，证据与溯源是主产出，生成文本只做解释层，这与 v1.0 的“生成优先”天然冲突。  
此外，v1.0 固定链式流程对用户任务的适配性差：研究者的需求是“先检索—再验证—再写作”，而不是“一次性生成”。  
另外，链式多 Agent 需要在每个阶段重复加载上下文，这会显著增加延迟与 Token 成本；而当前任务以检索与溯源为主，本质是工具路由与结果整合，一个主 Agent 更可控、成本更低、可解释性更强。  
因此我把 v1.0 定位为历史尝试，转向 v3.0 的检索与溯源闭环，能在面试里更清晰地证明系统价值：**可追溯、可验证、可复用**。

### v2.0 保留为基线的理由
- **可对照验证**：用于说明“多 Agent 编排并非最优的 MVP 路线”。
- **技术沉淀**：包含工具调度、解析尝试、RAGFlow 评估等记录。
- **失败成本低**：保留文档与脚本，不影响 v3.0 主线推进。

---

## 影响与边界

- **开发影响**：v3.0 作为唯一主线，所有新开发与接口设计以 v3.0 为准。
- **文档归档**：v1.0/v2.0 文档标注为历史记录，不在主线“文档导航”中优先展示。
- **代码复用限制**：除非明确确认可迁移，否则不复用 v1.0/v2.0 代码结构。

---

## 结论
- v1.0：弃用，仅保留历史记录。  
- v2.0：保留为实验/对比基线，不作为当前实现目标。  
- v3.0：唯一主线版本与交付目标。
