## 本阶段核心目标

## 架构决策回顾

### 为什么是 6 个 Agent？

| Agent | 职责 | 为什么独立 |
|-------|------|-----------|
| Planner | 分析章节结构 | 规划需要全局视角 |
| Query Generator | 生成检索查询 | 查询质量直接影响检索 |
| Retrieval | 执行检索 | 与 RAGFlow API 交互 |
| Ranker | 排序筛选 | 需要 LLM 判断相关性 |
| Writer | 生成文本 | 核心写作能力 |
| Verifier | 验证引用 | 质量把关，防止幻觉 |

**设计原则**：单一职责，便于调试和替换。

## 教训总结

### 1. 一个 Agent 一个文件

最初想把所有 Agent 放在一个文件里，结果：
- 文件太长，难以阅读
- 修改一个 Agent 容易影响其他
- 无法单独测试

**解决方案**：`agents/` 目录，每个 Agent 独立文件。

**认知标签**：模块化不是过度设计，是必要的工程实践

### 2. 先测试再集成

开发顺序：
1. 写 Agent 代码
2. 在 `if __name__ == "__main__"` 中测试
3. 测试通过后再写下一个
4. 全部通过后再写 workflow.py

如果跳过单独测试直接集成，出错时无法定位是哪个 Agent 的问题。

### 3. Pydantic 是救星

Agent 之间的数据传递用 Pydantic 模型定义：
- 类型安全，IDE 有提示
- 自动验证，错误提前暴露
- 文档自动生成

## 被放弃的方案

| 方案 | 放弃原因 |
|------|---------|
| 使用 LangChain Agent | 太重，不如直接用 LangGraph |
| Agent 间用 JSON 传递 | 类型不安全，容易出错 |
| 统一的 Agent 基类 | 过度抽象，每个 Agent 差异太大 |

---

## 遗留问题

1. **真实数据集验证**：目前测试用的是小数据集，需要用更多论文验证
2. **性能优化**：6 个 Agent 串行执行，速度较慢
3. **错误恢复**：某个 Agent 失败后，整个链条中断

这些问题在 v2.0 中通过 Orchestrator-Worker 架构解决。
