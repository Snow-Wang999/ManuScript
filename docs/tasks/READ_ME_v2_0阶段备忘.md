## 本阶段核心目标

**实现动态调度系统**：Orchestrator 根据章节复杂度分配不同 Worker，支持并行处理。

这是从"流水线"到"智能调度"的升级。

---

## 架构设计理念

### 为什么需要 Orchestrator-Worker？

v1.0 的问题：
1. **串行执行**：6 个 Agent 依次执行，速度慢
2. **一刀切处理**：简单章节和复杂章节用同样的流程
3. **单点失败**：一个 Agent 出错，整个链条中断

v2.0 的解决方案：
1. **并行处理**：多个章节同时处理
2. **复杂度适配**：简单章节用轻量 Worker，复杂章节用重量 Worker
3. **隔离失败**：单个任务失败不影响其他任务

## Worker 类型设计

| Worker | 适用章节 | 检索查询数 | 排序方式 |
|--------|---------|-----------|---------|
| Simple | 背景、结论、摘要 | 2 | 分数排序 |
| Complex | 方法、实验、结果 | 5 | LLM 辅助排序 |
| Review | 全局审核 | - | 引用验证 + 幻觉检测 |

**设计原则**：用最少的资源完成任务，复杂度匹配。

---

## 风险预判

| 风险 | 预判 | 应对措施 |
|------|------|---------|
| 并行导致 API 限流 | 中等 | Semaphore 控制并发数 (MAX=3) |
| 复杂度判断不准 | 低 | 可手动指定章节类型 |
| Review Worker 过于严格 | 中等 | 质量阈值可配置 |

---

## 与 v1.0 的关键区别

| 特性 | v1.0 | v2.0 |
|------|------|------|
| 架构 | 固定 Agent 链 | 动态任务分配 |
| 并行 | 串行执行 | 支持并行 (MAX=3) |
| 复杂度适配 | 统一处理 | 按复杂度选择 Worker |
| 全局审核 | 无 | Review Worker |
| 错误处理 | 链式失败 | 隔离任务失败 |
