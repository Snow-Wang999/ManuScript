### 1.2 演进策略

> **核心理念**：先跑通最小原型，再逐步增加复杂度，用真实的开发经历验证每个架构决策。

| 策略 | 说明 | 好处 |
|------|------|------|
| **渐进式** | 从简单到复杂，逐步验证 | 降低风险，避免方向性错误 |
| **问题驱动** | 只有当前版本有明确问题时才升级 | 避免过度设计 |
| **真实验证** | 每个版本都要实际开发和测试 | 决策基于真实数据而非臆测 |
| **记录导向** | 详细记录每次升级的原因和过程 | 形成可复盘的产品迭代故事 |

## 2. 核心设计原则

### 2.1 产品原则

在做架构决策时，我们遵循以下原则：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        ManuScript 产品原则                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 【聚焦原则】                                                       │
│      识别核心价值，外包非核心能力                                       │
│      → 我们的价值是"写作流程"，不是"PDF 解析"                          │
│                                                                         │
│   2. 【场景匹配原则】                                                   │
│      技术方案要匹配具体场景，而非追逐热点                               │
│      → Deep Research 适合探索，Material Assembly 适合学术写作           │
│                                                                         │
│   3. 【信任优先原则】                                                   │
│      高风险场景（学术、医疗、金融）中，信任度 > 效率                    │
│      → 宁愿多一步确认，也不要黑盒输出                                   │
│                                                                         │
│   4. 【可控性原则】                                                     │
│      AI 工具应增强用户能力，而非替代用户决策                            │
│      → 用户确认每一步，而非一键生成                                     │
│                                                                         │
│   5. 【简单性原则】                                                     │
│      能用简单方案解决的问题，不要用复杂方案                             │
│      → 如果 v0.2 够用，就不需要跳到 v2.0                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 技术原则

| 原则 | 说明 | 应用场景 |
|------|------|---------|
| **最小可验证** | 每个版本聚焦验证特定假设 | 版本规划 |
| **快速迭代** | 先跑通再优化，不追求完美 | 开发节奏 |
| **可观测** | 每个环节都要可调试可观测 | Agent 设计 |
| **容错性** | 单点故障不影响整体 | 错误处理 |

## 3. 关键技术决策

> 以下是项目开始前已明确的关键技术决策，不随版本迭代改变。

### 3.1 自建 RAG vs RAGFlow 托管

**决策**：使用 RAGFlow 托管

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     决策：RAGFlow 托管                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【背景】                                                               │
│  构建学术写作工具需要处理 PDF 解析、文本切分、向量检索等能力            │
│                                                                         │
│  【选项对比】                                                           │
│  ┌─────────────────┬─────────────────┬─────────────────┐               │
│  │ 维度            │ 自建 RAG        │ RAGFlow 托管    │               │
│  ├─────────────────┼─────────────────┼─────────────────┤               │
│  │ PDF 解析质量    │ 60-70%（需优化） │ 90%+            │               │
│  │ 工程投入        │ 高（3-6个月）    │ 低（API对接）   │               │
│  │ 维护成本        │ 持续投入         │ 几乎无          │               │
│  │ 灵活性          │ 完全可控         │ 受限于 API      │               │
│  └─────────────────┴─────────────────┴─────────────────┘               │
│                                                                         │
│  【决策理由】                                                           │
│  1. PDF 解析是"已解决的问题"，不是我们的核心价值                       │
│  2. RAGFlow 已投入大量工程优化，我们无法超越                            │
│  3. 节省 60% 工程量，聚焦写作流程创新                                   │
│                                                                         │
│  【风险缓解】                                                           │
│  • 实现重试和降级机制                                                   │
│  • 抽象 RAG 接口，支持未来替换                                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 Deep Research vs Material Assembly

**决策**：以 Material Assembly 为核心

```
┌─────────────────────────────────────────────────────────────────────────┐
│                 决策：Material Assembly 模式                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【两种模式对比】                                                       │
│                                                                         │
│  Deep Research：                                                        │
│  用户输入问题 → AI 自主搜索互联网 → 多轮迭代 → 生成报告                │
│  适合："我不知道有什么信息" → 帮我发现                                 │
│                                                                         │
│  Material Assembly：                                                    │
│  用户提供文献库 → AI 在封闭域检索 → 用户确认素材 → 生成草稿            │
│  适合："我知道信息在哪，但组织不起来" → 帮我组装                       │
│                                                                         │
│  【学术写作场景分析】                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │ 维度         │ Deep Research    │ Material Assembly │ 学术需求   │  │
│  ├─────────────────────────────────────────────────────────────────┤  │
│  │ 信息源       │ 开放网络         │ 用户文献库        │ 需可控来源 │  │
│  │ 幻觉率       │ 高（30-70%）     │ 低（<5%）         │ 零容忍     │  │
│  │ 引用可靠性   │ 不确定           │ 100% 可溯源       │ 必须准确   │  │
│  │ 用户控制     │ 低（黑盒）       │ 高（全程可见）    │ 人在回路   │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  【决策理由】                                                           │
│  学术写作者通常已读了大量论文，问题不是"信息不够"，                     │
│  而是"信息太多，组织不起来"                                            │
│                                                                         │
│  【演进计划】                                                           │
│  v3.0 考虑引入受控的 Deep Research 作为补充                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 一键生成 vs 分节渐进

**决策**：分节渐进生成

```
┌─────────────────────────────────────────────────────────────────────────┐
│                  决策：分节渐进生成                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【选项对比】                                                           │
│  ┌───────────────────┬───────────────┬───────────────┐                 │
│  │ 维度              │ 一键生成全文  │ 分节渐进生成  │                 │
│  ├───────────────────┼───────────────┼───────────────┤                 │
│  │ 用户体验          │ 看起来很酷    │ 需要多次操作  │                 │
│  │ 可控性            │ 极低          │ 高            │                 │
│  │ 幻觉风险          │ 极高          │ 低            │                 │
│  │ 用户心理负担      │ 高（要核对5000字）│ 低（逐段确认）│              │
│  └───────────────────┴───────────────┴───────────────┘                 │
│                                                                         │
│  【用户心理分析】                                                       │
│  "一键生成的 5000 字，我要花多久才能核对完？"                           │
│  "万一有一句引用是假的，整篇论文都废了"                                 │
│  "我宁愿一节一节写，至少心里有底"                                       │
│                                                                         │
│  结论：学术写作中，"控制感"比"便捷性"更重要                            │
│                                                                         │
│  【核心设计】                                                           │
│  1. 用户必须先确认大纲                                                  │
│  2. 每次只处理一个 Section                                              │
│  3. 先展示素材，用户确认后再生成                                        │
│  4. 用户可以随时编辑、重新生成                                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.4 Agent 架构选型

**决策**：从简单到复杂，渐进式选型

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Agent 架构渐进选型                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【架构选项】                                                           │
│                                                                         │
│  ① 单一 Prompt                   ② Prompt Chain                        │
│  ┌─────────────────────┐         ┌─────────────────────┐               │
│  │ 输入 ──► LLM ──► 输出│         │ P1 ──► P2 ──► P3 ──►│               │
│  └─────────────────────┘         └─────────────────────┘               │
│  简单但不可控                     可分步调试                            │
│                                                                         │
│  ③ Chain-of-Agents               ④ Orchestrator-Worker                 │
│  ┌─────────────────────┐         ┌─────────────────────┐               │
│  │ A1 ──► A2 ──► A3 ──►│         │     Orchestrator    │               │
│  │ (专业)(专业)(专业)  │         │    ╱    │    ╲      │               │
│  └─────────────────────┘         │  W1    W2    W3     │               │
│  专业化分工，可独立优化          └─────────────────────┘               │
│                                   动态调度，支持并行                    │
│                                                                         │
│  【渐进选型策略】                                                       │
│  ┌───────────┬───────────────┬─────────────────────────┐               │
│  │ 版本      │ 架构          │ 选型理由                │               │
│  ├───────────┼───────────────┼─────────────────────────┤               │
│  │ v0.1      │ 单一 Prompt   │ 最快验证 RAGFlow 连通性 │               │
│  │ v0.2      │ Prompt Chain  │ 验证分节生成可行性      │               │
│  │ v1.0      │ Chain-of-Agents│ 验证专业化 Agent 效果  │               │
│  │ v2.0      │ Orchestrator  │ 如需并行和灵活调度      │               │
│  └───────────┴───────────────┴─────────────────────────┘               │
│                                                                         │
│  【原则】不预设最终架构，根据实际问题决定是否升级                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```
